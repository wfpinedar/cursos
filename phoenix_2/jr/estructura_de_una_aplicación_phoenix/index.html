Estructura de una aplicación Phoenix¶

Una aplicación Phoenix está basada en el patrón de arquitectura MVC (Modelo-Vista-Controlador) y sigue una estructura jerárquica bien definida, lo que facilita su mantenimiento y escalabilidad.

Explicación teórica¶

La estructura de una aplicación Phoenix consta de varios directorios y archivos importantes que desempeñan un papel clave en su funcionamiento. A continuación, se explican brevemente cada uno de ellos:

  • config: Este directorio contiene todos los archivos de configuración de la aplicación, como por ejemplo config.exs, que contiene la configuración general de la aplicación, y dev.exs y prod.exs, que contienen la configuración específica para el entorno de desarrollo y producción respectivamente.

  • lib: En este directorio se encuentran los módulos de la aplicación, que contienen la lógica de negocio y la manipulación de datos.

  • test: Aquí se encuentran los tests de la aplicación, escritos en el framework de testing ExUnit.

  • web: Este directorio es el corazón de la aplicación Phoenix y contiene todos los archivos relacionados con la interfaz web. A su vez, se divide en subdirectorios:

  • controllers: Contiene los controladores de la aplicación, que se encargan de recibir las solicitudes HTTP y enviar las respuestas correspondientes.

  • models: En este directorio se encuentran los modelos de la aplicación, que representan las entidades de la base de datos y contienen la lógica de acceso a los datos.

  • views: Aquí se encuentran las vistas de la aplicación, que son responsables de presentar los datos al usuario en formato HTML o algún otro formato.

  • templates: Contiene los archivos de plantilla utilizados para generar las vistas de la aplicación.

  • channels: Este directorio contiene los canales de comunicación en tiempo real, que permiten la interacción en tiempo real entre el servidor y los clientes a través de WebSockets.

  • static: Aquí se encuentran los archivos estáticos de la aplicación, como hojas de estilo, imágenes y archivos JavaScript.

  • errors: Contiene los archivos de error personalizados, que se muestran cuando ocurre un error en la aplicación.

Palabras clave y su definición¶

  • MVC: Acrónimo de Modelo-Vista-Controlador, es un patrón de arquitectura de software que divide una aplicación en tres componentes principales: el Modelo, que se encarga de la lógica de negocio y la manipulación de datos; la Vista, que se encarga de presentar los datos al usuario; y el Controlador, que se encarga de recibir las solicitudes del usuario y enviar las respuestas correspondientes.

  • Interfaz web: Es la parte de una aplicación que interactúa con el usuario a través del navegador web.

  • Framework de testing: Es una herramienta que proporciona un conjunto de funciones y métodos para facilitar la creación y ejecución de tests automatizados.

  • ExUnit: Es el framework de testing utilizado en Phoenix Framework, que permite escribir tests en un estilo simple y declarativo.

  • WebSockets: Son un protocolo de comunicación bidireccional entre un navegador web y un servidor, que permite la comunicación en tiempo real.

Ejemplos de código en Phoenix Framework¶

Controlador¶

defmodule MyApp.UserController do
  use MyApp.Web, :controller

  def index(conn, _params) do
    users = MyApp.User |> MyApp.Repo.all
    render(conn, "index.html", users: users)
  end

  def show(conn, %{"id" => id}) do
    user = MyApp.User |> MyApp.Repo.get!(id)
    render(conn, "show.html", user: user)
  end
end

Vista¶

defmodule MyApp.UserView do
  use MyApp.Web, :view

  def render("index.html", %{users: users}) do
    ~H"""
    <ul>
      <%= for user <- users do %>
        <li><%= user.name %></li>
      <% end %>
    </ul>
    """
  end

  def render("show.html", %{user: user}) do
    ~H"""
    <h1><%= user.name %></h1>
    <p><%= user.email %></p>
    """
  end
end

Ejercicios prácticos con instrucciones claras¶

  1. Crea una nueva aplicación Phoenix con el nombre "MyApp".
  2. Agrega un nuevo modelo llamado "Task", con los campos "title", "description" y "completed".
  3. Genera un controlador para el modelo Task con las acciones de índice y mostrar.
  4. Actualiza el archivo router.ex para agregar las rutas correspondientes a las acciones del controlador.
  5. Crea una vista para el modelo Task, que muestre una lista de tareas en la acción de índice y los detalles de una tarea en la acción de mostrar.
  6. Crea una plantilla de error personalizada para el error 404.
  7. Ejecuta la aplicación y comprueba que todo funcione correctamente.

Consejos o mejores prácticas¶

  • Sigue la convención de nombres de Phoenix para mantener una estructura coherente en tu aplicación.
  • Utiliza los subdirectorios de web para organizar tu código de manera lógica y fácilmente mantenible.
  • Utiliza los archivos config.exs, dev.exs y prod.exs para separar la configuración general de la específica para cada entorno.
  • Utiliza el framework de testing ExUnit para escribir tests automatizados y asegurarte de que tu aplicación funciona correctamente.
  • Aprovecha las facilidades que ofrece Phoenix, como la generación automática de código y la recarga automática en desarrollo, para aumentar tu productividad.
