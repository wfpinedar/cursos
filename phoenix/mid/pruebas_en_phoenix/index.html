Pruebas en Phoenix¶

Descripción del módulo¶

En este módulo aprenderemos sobre la importancia de las pruebas en el desarrollo de aplicaciones con Phoenix Framework. Veremos cómo utilizar herramientas como ExUnit y Hound para escribir pruebas automatizadas que nos ayuden a garantizar el correcto funcionamiento de nuestra aplicación.

Explicación teórica¶

Las pruebas son una parte fundamental en el desarrollo de aplicaciones. Nos permiten verificar que nuestro código funciona correctamente y que no se introducen errores al realizar cambios o añadir nuevas funcionalidades. Además, nos ayudan a mantener un código más limpio y organizado, ya que nos obligan a escribir un código más modular y fácil de probar.

En Phoenix Framework, las pruebas se realizan a través de dos herramientas principales: ExUnit y Hound. ExUnit es una librería de pruebas unitarias que nos permite probar funciones y módulos de forma aislada. Por otro lado, Hound es una librería de pruebas de integración que nos permite simular interacciones con la aplicación como si fuera un usuario real.

Palabras clave y definiciones¶

  • Pruebas: Son el proceso de verificar que nuestro código funciona correctamente.
  • ExUnit: Librería de pruebas unitarias en Phoenix Framework.
  • Hound: Librería de pruebas de integración en Phoenix Framework.
  • Pruebas unitarias: Son pruebas que se realizan sobre funciones y módulos de forma aislada.
  • Pruebas de integración: Son pruebas que se realizan sobre la aplicación en su conjunto, simulando interacciones con ella.

Ejemplos de código en Phoenix Framework¶

Pruebas unitarias con ExUnit¶

defmodule CalculatorTest do
  use ExUnit.Case

  test "suma dos números" do
    assert Calculator.sum(2, 3) == 5
  end

  test "resta dos números" do
    assert Calculator.sub(5, 2) == 3
  end
end

Pruebas de integración con Hound¶

defmodule LoginTest do
  use ExUnit.Case, async: true
  use Hound.Helpers

  setup do
    :ok = Application.ensure_all_started(:my_app)
    {:ok, driver: "phantomjs"}
  end

  test "iniciar sesión con credenciales válidas" do
    navigate_to("http://localhost:4000/login")
    fill_field("#email", "test@example.com")
    fill_field("#password", "password")
    click("#login-btn")
    assert_title("Página de inicio")
  end
end

Consejos o mejores prácticas¶

  • Es importante tener una buena cobertura de pruebas, pero no es necesario probar absolutamente cada línea de código. Enfócate en las partes críticas y en las funcionalidades más complejas.
  • Utiliza nombres descriptivos para tus pruebas, esto te ayudará a entender rápidamente qué se está probando.
  • No te olvides de actualizar tus pruebas cuando hagas cambios en el código. De lo contrario, podrías estar probando funcionalidades que ya no existen o introduciendo nuevos errores.
  • Utiliza herramientas de análisis de cobertura de pruebas para asegurarte de que no hay partes de tu código que no están siendo probadas.
  • Siempre piensa en los casos de prueba más extremos y en los posibles errores que pueden ocurrir.
  • No tengas miedo de refactorizar tus pruebas si el código cambia y se vuelve más fácil de probar. Las pruebas también deben evolucionar junto con el código.
